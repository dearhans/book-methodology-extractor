# Book Methodology Extractor

从书籍中提取核心方法论，为每个方法论生成一个独立的 Hermes Agent 技能。

## 工作流程

### 1. 接收书籍信息

支持三种方式：

**方式 1: 书籍名称（网络搜索）**
- 书籍完整名称
- 作者（如果有多个同名书籍）
- 是否有特定的方法论需要提取（可选）

**方式 2: 上传文件（更准确）**
- 支持 PDF、Word、TXT、Markdown 等格式
- 直接从文件内容提取方法论
- 更准确，避免网络信息偏差
- 适合有电子书的用户

**方式 3: GitHub 自动化发布**
- 自动生成完整的项目文档和结构
- 创建 README、示例、模板等
- 支持一键式 GitHub 仓库创建和推送
- 包含 MIT 许可证和 .gitignore

**推荐：** 如果用户有书籍文件，优先使用文件上传方式。如果需要分享或版本控制，使用 GitHub 自动化发布。

### 2. 提取书籍方法论

#### 方式 A: 从文件提取（推荐）

如果用户上传了文件：

1. **读取文件内容**
   - 使用 `read_file` 工具读取上传的文件
   - 支持 PDF、Word、TXT、Markdown 等格式

2. **识别核心章节**
   - 查看目录结构
   - 识别包含方法论的章节
   - 重点关注：框架、模型、步骤、方法论等关键词

3. **提取方法论**
   - 方法论名称（中英文）
   - 方法论的核心目的和适用场景
   - 方法论的关键步骤或框架
   - 实际应用案例或示例
   - 相关的斜杠命令建议

#### 方式 B: 网络搜索

如果没有文件，使用 `search_files` 结合网络信息（通过 terminal + curl/wget）搜索书籍的核心方法论、框架或模型。搜索策略：

**搜索关键词组合:**
- "{书名} 核心方法论"
- "{书名} 主要观点"
- "{书名} 方法论总结"
- "{书名} 实践框架"
- "{书名} 关键概念"

**信息来源优先级:**
1. 官方书籍介绍和目录
2. 权威书评和读书笔记
3. 作者公开演讲或文章
4. 知名平台的书摘（如豆瓣、知乎、微信公众号）

**提取要点:**
- 方法论名称（中英文）
- 方法论的核心目的和适用场景
- 方法论的关键步骤或框架
- 实际应用案例或示例
- 相关的斜杠命令建议（如 /mvp）

#### 方式 C: GitHub 自动化发布

当用户需要分享或版本控制提取的方法论时：

1. **创建项目结构**
   - README.md（项目说明）
   - SKILL.md（核心技能定义）
   - examples/（使用示例）
   - templates/（技能模板）
   - LICENSE（MIT 许可证）
   - .gitignore（Git 忽略配置）

2. **生成文档**
   - 自动编写 README 包含功能说明
   - 生成使用示例和最佳实践
   - 创建技能模板库
   - 添加 GitHub 自动化脚本

3. **初始化 Git 仓库**
   - 配置用户信息
   - 创建初始提交
   - 设置远程仓库
   - 提供推送脚本

### 3. 处理中断和停止信号

**重要工作流程规则：**

当用户发出"停止"、"暂停"或类似信号时：

1. **立即暂停当前操作**
   - 停止生成内容或执行任务
   - 保存当前进度状态
   - 确认已暂停

2. **记录中断点**
   - 标记已完成的工作
   - 记录下一步计划
   - 确保可以从断点恢复

3. **等待用户指示**
   - 询问是否继续、修改方向或重新开始
   - 尊重用户的选择
   - 不要假设用户意图

4. **恢复执行**
   - 从断点继续，避免重复工作
   - 如有方向调整，按新要求执行
   - 确保最终输出完整

**内容生成偏好：**
- 优先简洁、可执行的内容
- 避免过度解释和冗长描述
- 定期检查用户是否需要调整
- 提供工作代码优先，理论说明其次

### 4. 整理方法论列表

将搜索到的信息整理成结构化的方法论列表，包含：

```json
{
  "book_name": "书籍名称",
  "author": "作者",
  "methodologies": [
    {
      "name_en": "minimum-viable-product",
      "name_cn": "最小可行产品",
      "trigger_scenario": "准备开发产品，想用最低成本验证核心假设",
      "description": "用最简单的产品版本验证核心假设，快速获得市场反馈",
      "steps": ["确定核心假设", "设计最小功能集", "开发MVP", "测试验证", "收集反馈"],
      "output_format": "包含假设验证、功能清单、开发计划的完整方案",
      "examples": "Dropbox早期用视频演示验证需求，避免开发无用功能"
    }
  ]
}
```

### 5. 为每个方法论生成 Hermes Agent Skill

为每个方法论创建一个独立的 skill 文件夹，结构如下：

```
{methodology-name}/
├── SKILL.md          # 技能定义
└── examples/         # 使用示例（可选）
    └── example.md
```

**SKILL.md 模板:**

```markdown
---
name: {methodology-name}
description: {触发场景描述}。当用户{具体触发条件}时使用此技能。
category: knowledge
---

# {方法论中文名称}

{方法论的详细描述，包括来源书籍、核心理念等}

## 来源

## 适用场景

- {场景1}
- {场景2}
- {场景3}

## 执行步骤

### 步骤 1: {步骤名称}
{详细说明}

### 步骤 2: {步骤名称}
{详细说明}

### 步骤 3: {步骤名称}
{详细说明}

## 输出格式

{描述最终输出的格式和内容}

**示例输出:**

```
{提供一个具体的输出示例}
```

- {注意事项1}
- {注意事项2}

```

### 6. 打包和交付

将所有生成的 skill 文件放在统一的输出目录中，并提供：

1. **方法论清单文档** - 列出所有提取的方法论及其对应的 skill
2. **多个独立的 skill 文件** - 每个方法论一个可安装的 skill
3. **安装说明** - 如何在 Hermes Agent 中安装和使用这些 skill
4. **GitHub 项目**（可选） - 完整的版本控制和分享方案

## 输出交付格式

最终向用户提供：


## 法律与伦理边界

### 服务条款合规
- **严禁**创建违反第三方服务条款的自动化技能
- **严禁**绕过安全机制或进行未经授权的系统访问
- **优先**使用官方API、SDK或授权机制
- **必须**明确告知用户潜在的法律/服务条款风险

### 隐私与安全
- **绝不**存储或处理用户凭证、API密钥、密码等敏感信息
- **必须**使用 `[REDACTED]` 占位符处理敏感数据
- **优先**处理用户提供的导出数据而非尝试自动化访问
- **明确**告知数据处理的隐私影响

### 替代方案优先
当用户请求可能违规的功能时：
1. **明确拒绝**违规方案并解释原因
2. **提供**官方/合法的替代方案
3. **建议**使用官方导出/导入功能
4. **推荐**手动处理而非自动化访问

## 质量标准

**方法论提取质量:**
- 准确反映书籍原意，不曲解作者观点
- 方法论之间有清晰边界，不重复
- 每个方法论都有明确的适用场景
- 提供可执行的步骤，而非抽象概念
- **包含法律/伦理注意事项（如适用）**

**Skill 生成质量:**
- description 清晰准确，便于触发
- 适用场景具体，用户容易理解何时使用
- 执行步骤可操作，有明确的输入输出
- **简洁明了，避免过度解释**
- 包含示例，帮助用户理解预期结果
- **明确标注合规边界和限制**

**文档质量:**
- README 清晰说明项目用途和安装方法
- 示例代码可直接运行或修改
- 模板易于定制和扩展
- 文档语言简洁，重点突出

## 特殊情况处理

**文件上传相关：**
- 如果文件格式不支持，提示用户支持的格式（PDF、Word、TXT、Markdown）
- 如果文件过大或内容过多，优先提取目录和核心章节
- 如果文件内容不清晰，建议用户提供书名进行网络搜索补充

**书籍信息不足：**
- 如果搜索结果不足以提取完整方法论，明确告知用户
- 提供已找到的部分信息，询问用户是否继续
- 建议用户上传书籍文件或提供更多上下文

**方法论过多:**
- 如果书籍包含大量方法论（>15个），优先提取最核心的 8-12 个
- 询问用户是否有特定关注的方法论类型
- 提供方法论清单供用户选择

**方法论名称冲突:**
- 如果方法论名称已被占用，添加书籍前缀（如 "lean-startup-mvp"）
- 确保每个 skill 名称唯一且有意义

**用户中断处理:**
- 立即停止当前操作
- 保存进度并标记断点
- 询问用户后续意图
- 从断点恢复或按新要求执行

## 工具使用指南

### 主要工具
- `search_files` - 搜索本地文件内容
- `terminal` - 执行网络搜索和文件操作
- `read_file` - 读取上传的文件
- `skill_manage` - 生成 Hermes Agent 技能
- `write_file` - 创建输出文档

### 工作流命令示例

```bash
# 搜索书籍信息
terminal("curl -s 'https://api.example.com/books?q={书名}'")

# 读取上传文件
read_file("/path/to/book.pdf")

# 生成 skill
skill_manage(
    action='create',
    name='mvp',
    category='knowledge',
    content=skill_content
)
```

## 最佳实践

1. **优先使用文件上传** - 更准确，避免网络信息偏差
2. **验证信息准确性** - 交叉验证从不同来源获取的信息
3. **保持方法论完整性** - 确保提取的方法论完整且可执行
4. **提供实用示例** - 每个方法论都包含实际应用案例
5. **测试生成的 skill** - 确保生成的 skill 可以正常工作
6. **尊重用户中断** - 立即响应停止信号，保存进度
7. **简洁输出优先** - 避免冗长解释，提供可执行内容

## 限制和注意事项

- 网络搜索依赖外部 API 和网站的可用性
- PDF 解析可能因文件格式而异
- 复杂书籍可能需要人工审核提取结果
- 生成的 skill 需要用户确认后再安装
- 版权问题：仅提取公开的、非版权保护的方法论和概念
- GitHub 操作需要用户明确授权和认证
